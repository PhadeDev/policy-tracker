Document Link Modal — Manual Rebuild Guide

This replaces the previous version of this guide entirely. The previous approach tried to paste the whole redesigned modal in as one block of YAML — Studio rejected it (PA1001 / YamlInvalidSyntax) because a single-control YAML fragment pasted from outside Studio isn't something its clipboard-paste feature reliably accepts; it expects either a full Screens: document or a control it copied itself. That's on the guide, not on anything you did.

Nothing here needs a paste of any kind. Every step below is either: select a control already in your app, change one property in the formula bar, or insert one new control from the Insert panel and set a couple of properties on it by hand. Nothing is deleted that doesn't already need deleting (nothing does, as it turns out), and nothing needs renaming except the two brand-new controls in Part 2.

You already have ctnLinkModal_2 back (the version from before tonight's edits). That means the two real bugs from earlier — the display name field getting silently overwritten, and stale text showing up after Cancel — are still present in what you have right now. Part 1 below fixes those two things. Do Part 1 tonight regardless of anything else. Part 2 (visual restyle to match the rest of the screen) is lower priority — nice to have before the demo, not required for it to work correctly.

Part 1 — Fix the two real bugs ~10 minutes, zero risk

Every change below is a property edit on a control that already exists. No inserting, no deleting, no renaming.

1a. Display name field gets silently overwritten

What's wrong: the "Link Display Name" box falls back to a filename auto-guessed from the URL field whenever a context variable called DisplayNamePrefill changes — which happens live as you type the URL, even after you've already typed your own name in first.

Where to clickWhat to do
Open ViewItem → open any item → click Add Link (or Edit on an existing link) to open the modal → in the Tree view, expand into ctnLinkModal_2 → cardLinkModal_2 → frmLink_2 → LinkText_DataCard2_2 and click DataCardValue28_2 (the actual text input, not the data card or its label). In the Properties dropdown at the top of the formula bar, select Default. Select all in the formula bar (Ctrl+A), delete, and type exactly: 
Parent.Default

1b. URL field OnChange (the thing that was pushing the bad value in) — remove it

The Default fix above stops DataCardValue28_2 from listening to DisplayNamePrefill, but the URL field is still setting that variable on every keystroke for no reason now. Clean it out so nothing's left half-wired.

Where to clickWhat to do
Still inside the modal's Tree: frmLink_2 → LinkUrl_DataCard2_2, click DataCardValue29_2 (the URL text input). In the Properties dropdown, select OnChange. Select all in the formula bar (Ctrl+A) and press Delete, leaving it completely blank. Click away from the control to commit the blank formula.

1c. Stale text left over after Cancel, then Add

What's wrong: when you open the modal fresh (via Edit or Add), Power Apps doesn't automatically clear the text boxes just because the underlying record changed — you have to explicitly tell the form to refresh with ResetForm(frmLink_2). Only the Cancel button does this today. The three places that open the modal don't, so whatever was last typed/shown just sits there.

Where to clickWhat to do
On the links list, click the Edit icon (pencil) on any existing link row to select EditLink_2 in the tree. Select OnSelect in the Properties dropdown. Find the line Set(varShowLinkModal, true) at the end and add a new line directly above it reading: 
ResetForm(frmLink_2);
So the formula ends with: 
Set(varSelectedLink, LookUp(PolicyLinks, ID = ThisItem.ID));
ResetForm(frmLink_2);
Set(varShowLinkModal, true)
Select btnNewLink_1 (the "Add Document Link" pill button near the link list). Same fix: in OnSelect, find Set(varSelectedLink, Defaults(PolicyLinks)); followed by Set(varShowLinkModal, true), and insert the ResetForm line between them: 
Set(varSelectedLink, Defaults(PolicyLinks));
ResetForm(frmLink_2);
Set(varShowLinkModal, true)
Select btnAddLinkEmpty1 (the "Add Document Link" button shown when the link list is empty). This one has two spots with the same Set(varSelectedLink, Defaults(PolicyLinks)); Set(varShowLinkModal, true) pair inside its OnSelect — one near the top of the formula, one further down inside the nested save-then-open-modal branch. Add the same ResetForm(frmLink_2); line between the two Set(...) calls in both spots.
Test all three: (1) type a display name, then paste a SharePoint URL — the name you typed should stay put. (2) Edit a link, note its text, hit Cancel, then click Add — the Add form should be completely blank, not showing the link you just cancelled out of.

Part 2 — Visual restyle to match the rest of the screen Optional

This brings the look in line with the Duplicate/Delete popups elsewhere on this screen (dark dimmed background, rounded card, a proper title + close button) — but it's cosmetic. Skip this section entirely tonight if you're short on time; Part 1 is what actually matters for the demo working correctly.

To keep this low-risk, nothing gets deleted or converted. The modal stays a ManualLayout container (converting it to AutoLayout would mean deleting and rebuilding the whole thing from scratch, which is exactly the kind of risk we're avoiding tonight) — instead we restyle the existing pieces in place and add two small new controls for the title bar.
