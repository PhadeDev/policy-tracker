Pre-Release Scrub — Save/Delete/Duplicate Integrity Pass

Targeted scrub of every Patch / Remove / SubmitForm path in the app, looking specifically for ways a save, delete, or duplicate could fail silently or fire twice. Four issues found, all fixed in the local files already — this guide is just what to paste into Studio. In priority order:

Jump to: 1. Main — Publish buttons show "success" even when the write fails Highest risk, on the demo screen 2. ViewItem — Duplicate button has a double-click window that can create two copies Real bug 3. ViewItem — Delete-link button never actually disables during delete Minor 4. ViewItem — Add/Edit Link form has no failure handling at all Real bug 5. ViewItem — Document Link modal restyled to match the rest of the new design Cosmetic, whole control

1. Main — Publish buttons show "success" even when the write fails Highest risk, on the demo screen

What was wrong: both the single "⚡ Publish" button and "Publish All" ran their Patch(...) calls with no error checking whatsoever. If a Patch failed for any reason (dropped connection, a permissions blip, SharePoint hiccup), the code carried straight on to Refresh and showed a green "Published successfully" toast anyway — telling you it worked when it didn't. For Publish All specifically, this is worse: it loops over every item in a week's group, and if 3 of 5 patch calls fail, you're told all 5 succeeded with no way to know which two didn't actually go through.

Update after testing (two rounds): the first version of the Publish All fix counted failures with Set(varPublishFailCount, varPublishFailCount + 1) run from inside the ForAll loop. Studio rejected that on paste with "This function cannot be invoked within ForAll" — that's a real, current Power Apps platform rule, not a mistake in the formula: ForAll iterations aren't guaranteed to run in strict order, so Microsoft blocks Set() on a global variable from inside the loop to avoid race conditions. Swapping to Collect() into a new collection colPublishFailures fixed that, but caused a second, different error: "Invalid argument type (Table). Expecting a Record value instead." That one's a type conflict inside IfError — Patch(...) returns a Record, but Collect(...) returns the whole Table it just collected into, and IfError's two branches (the normal result and the error fallback) have to resolve to the same type. Fix: end both branches with ; true, so both sides evaluate to a plain Boolean instead of their natural Record/Table return values — this is the same trick already used safely elsewhere (a branch ending in a chain of statements just returns whatever the last statement evaluates to). The block below has both fixes applied.
Where to click (1st block): open the Main screen. In the Tree view, find BossPublishBtn (the small green "⚡ Publish" button on each item row — only visible to boss-button users) and click it directly. Select OnSelect from the Properties dropdown, select all (Ctrl+A), delete, paste the block below.

Full replacement for BossPublishBtn.OnSelect:

Set(
    varQuickPublishDate,
    Coalesce(ThisItem.'Planned Publish Date', Today())
);
IfError(
    Patch(
        'Policy Proof Tracker',
        LookUp('Policy Proof Tracker', ID = ThisItem.ID),
        {
            Published: true,
            'Published Date': varQuickPublishDate
        }
    ),
    Notify("Publish failed. Please check your connection and try again.", NotificationType.Error),
    Refresh('Policy Proof Tracker');
    RefreshActiveCache();
    Notify(
        "Marked as published - " & Text(varQuickPublishDate, "d mmm yyyy"),
        NotificationType.Success
    )
)
Where to click (2nd block): still on Main, find PublishAllBtn (the "Publish All (N)" button at the top of each week's group) and click it directly. Select OnSelect, select all, replace.

Full replacement for PublishAllBtn.OnSelect — now counts failures and tells you if any items didn't go through:

// PublishAllBtn.OnSelect
Set(varPublishGroupDate, ThisItem.PlannedDate);
Set(varPublishGroupCount, CountRows(Filter(ThisItem.GroupedItems, 'External consultation required' = false)));
Clear(colPublishFailures);
ForAll(
    Filter(ThisItem.GroupedItems, 'External consultation required' = false) As PubItem,
    IfError(
        Patch(
            'Policy Proof Tracker',
            LookUp('Policy Proof Tracker', ID = PubItem.ID),
            {
                Published: true,
                'Published Date': varPublishGroupDate
            }
        ); true,
        Collect(colPublishFailures, {ID: PubItem.ID}); true
    )
);
Set(varPublishFailCount, CountRows(colPublishFailures));
Refresh('Policy Proof Tracker');
RefreshActiveCache();
If(
    varPublishFailCount = 0,
    Notify(
        "Published all " & varPublishGroupCount & " items for " & Text(varPublishGroupDate, "d mmm yyyy"),
        NotificationType.Success
    ),
    Notify(
        varPublishFailCount & " of " & varPublishGroupCount & " items failed to publish for " & Text(varPublishGroupDate, "d mmm yyyy") & ". Please retry.",
        NotificationType.Error
    )
)
Test: publish a single item normally, confirm the green toast and that it moves out of the unpublished list. Can't easily fake a network failure to test the error path directly — the important thing is it no longer silently lies about success if one ever does fail.

2. ViewItem — Duplicate button has a double-click window that can create two copies Real bug

What was wrong: the Duplicate confirm button set its own "busy" flag (varDuplicating) back to false immediately after the Patch call finished, but before checking whether it actually succeeded and before closing the modal. That leaves a short window where the button re-enables itself while the modal is still open. A fast double-click in that window fires a second Patch — creating two duplicate entries instead of one. Also tightened the error check to look at the whole table's errors (same pattern used everywhere else in the app for Save) rather than a check scoped to the specific new record, which is the safer of the two ways to detect a failed write.

Where to click: open ViewItem, open any real item, click Duplicate (the copy icon top-right) to open the Duplicate modal, then in the Tree view find btnDuplicateConfirm1 inside DuplicateModal1 and click it directly. Select OnSelect from the Properties dropdown, select all, replace.

Full replacement for btnDuplicateConfirm1.OnSelect:

Set(varDuplicating, true);
Set(
    varNewDuplicate,
    Patch(
        'Policy Proof Tracker',
        Defaults('Policy Proof Tracker'),
        {
            Title: varSelectedRelease.Title,
            'Document Type': varSelectedRelease.'Document Type',
            'Item summary': varSelectedRelease.'Item summary',
            Owner: varSelectedRelease.Owner,
            Audience: varSelectedRelease.Audience,
            'External consultation details': varSelectedRelease.'External consultation details',
            'External consultation required': varSelectedRelease.'External consultation required',
            Remarks: varSelectedRelease.Remarks,
            'Title for Newsletter': varSelectedRelease.'Title for Newsletter',
            'High Priority': varSelectedRelease.'High Priority',
            'Reason for High Priority': varSelectedRelease.'Reason for High Priority',
            'Top Item': varSelectedRelease.'Top Item',
            'Planned Publish Date': varSelectedDuplicateDate,
            Published: false,
            'Published Date': Blank(),
            'Approved by Pillar Lead (Ready to Release)': false,
            'Approved For Release By': Blank(),
            EmailSent: false
        }
    )
);
If(
    IsEmpty(Errors('Policy Proof Tracker')),
    Set(varDuplicating, false);
    Set(varShowDuplicateModal1, false);
    ClearCollect(colUnpublishedPolicies, Filter('Policy Proof Tracker', Published = false));
    Notify(
        "Duplicated to " & LookUp(colThursdays, ThursdayDate = varSelectedDuplicateDate).ThursdayLabel & ". Find it on the main screen.",
        NotificationType.Success
    ),
    Set(varDuplicating, false);
    Notify(
        "Error: " & First(Errors('Policy Proof Tracker')).Message,
        NotificationType.Error
    )
)
Test: duplicate a real item to a future Thursday, confirm exactly one new entry appears on Main for that date, not two. Also try clicking Duplicate twice quickly to confirm the button now stays disabled the whole way through.

3. ViewItem — Delete-link confirm button never actually disables during delete Minor

What was wrong: the "Delete" confirm button in the delete-link popup checked varDeletingItem to decide whether to disable itself — but that variable belongs to the whole-item delete flow, not link deletion, and is never set by this button. So the button was never actually disabled while a link delete was in progress, and a fast double-click could fire Remove(PolicyLinks, ...) twice. The second call fails harmlessly since the link's already gone, but it throws an unnecessary error toast. Fixed by introducing a real varDeletingLink flag used only by this flow.

Where to click: open ViewItem, open any item with at least one document link, click the link's Delete icon to open the delete-link popup, then in the Tree view find btnModalAction1_1 inside that popup and click it directly. You'll need to change two properties on this control:

1. Select DisplayMode from the Properties dropdown, select all, replace with:

If(varDeletingLink, DisplayMode.Disabled, DisplayMode.Edit)

2. Select OnSelect, select all, replace with:

Set(varDeletingLink, true);
IfError(
    Remove(PolicyLinks, varSelectedLink),
    Set(varDeletingLink, false);
    UpdateContext({locDisplayAttDeletePopUp: false});
    Notify("Delete failed. Please check your connection and try again.", NotificationType.Error),
    Set(varLoadingLinks, true);
    BuildLinksForSource(varSelectedRelease.ID);
    Set(varLoadingLinks, false);
    Set(varDeletingLink, false);
    UpdateContext({locDisplayAttDeletePopUp: false});
    Set(varSelectedLink, Blank());
    Notify("Link deleted.", NotificationType.Success)
)
Test: delete a document link, confirm the toast and that it disappears from the list. Not a big deal if you skip this one before the demo — cosmetic edge case only.

4. ViewItem — Add/Edit Link form has no failure handling at all Real bug

What was wrong: the link form (frmLink_2) only has an OnSuccess property wired up — there's no OnFailure. If SubmitForm fails (bad connection, permissions issue), the user gets zero feedback: no error toast, the modal just stays open with no explanation. They'd have no way to know the link wasn't actually saved short of noticing it themselves.

Where to click: open ViewItem, open any item, click Add Link (or Edit on an existing link) to open the link modal, then in the Tree view click on frmLink_2 directly (the form itself, not one of its data cards). In the Properties dropdown, click the small + / "Add property" control if OnFailure isn't already listed, and add it. Select all in that property's formula bar and paste the line below.

frmLink_2.OnFailure (new property):

Notify("Save failed. Please check your connection and try again.", NotificationType.Error)
Test: save a link normally, confirm nothing changed about the success path. Can't easily force a failure to test directly, but the important part is a real error message now exists where there was silence before.
Everything else checked, no further issues found: the main Save/Save & Close flows on ViewItem already check the whole table's Errors() and show a proper failure toast — those are fine as-is. The item-level Delete flow already wraps its Remove() in IfError correctly. Nothing else in Main, Historic, Overview, or NewsletterPack writes to the data source at all — they're read-only screens, so there's no equivalent risk there.

5. ViewItem — Document Link modal restyled to match the rest of the new design

What changed: the "Add/Edit Document Link" popup (ctnLinkModal_2) was the one part of ViewItem still on the old plain-white, sharp-corner, no-overlay look, while every other popup on this screen (Duplicate, Delete item, Delete link, Discard changes) already uses the dark dimmed-overlay + rounded centred card style. This brings it in line: dark overlay, rounded card, a proper title bar with an "Add Document Link" / "Edit Document Link" heading that switches automatically plus a ✕ close button, ALL-CAPS field labels, and Cancel/Save buttons restyled to match the Duplicate modal's grey/dark-green pair. The two bugs fixed above (free-typed display name, and the stale-text-after-cancel fix) are both already baked into this version.

This one is different from the property pastes above — it's a whole control, not a single formula. It's about 450 lines of YAML replacing the entire ctnLinkModal_2 container, not a value you paste into one property's formula bar. A plain copy-paste box (like the ones above) risks corrupting line endings on a paste this size, which is exactly the failure mode pa-yaml-wrap exists to avoid for full-screen exports. The button below uses that same safe method (forces real CRLF line endings) scoped to just this one control, so it can live right here in the guide instead of a separate file.
Where to click: open ViewItem. In the Tree view, expand down to find ctnLinkModal_2 (it sits inside a container called Attachments_1). Click ctnLinkModal_2 and delete it. Then click Attachments_1 (the parent) to select it, click the button below to copy, and press Ctrl+V with Attachments_1 selected — this inserts the new ctnLinkModal_2 as its child again.
Click to copy — CRLF-safe, ready for Studio paste
Test: after pasting, click "Add Document Link" — confirm the dark overlay, rounded card, "Add Document Link" title, and that the display name field is empty (not carrying over a previous link's text). Edit an existing link and confirm the title reads "Edit Document Link" and both fields are pre-filled correctly. Click the ✕ and confirm it closes cleanly same as Cancel.
