Document Link Modal — Manual Rebuild Guide

This replaces the previous version of this guide entirely. The previous approach tried to paste the whole redesigned modal in as one block of YAML — Studio rejected it (PA1001 / YamlInvalidSyntax) because a single-control YAML fragment pasted from outside Studio isn't something its clipboard-paste feature reliably accepts; it expects either a full Screens: document or a control it copied itself. That's on the guide, not on anything you did.

Nothing here needs a paste of any kind. Every step below is either: select a control already in your app, change one property in the formula bar, or insert one new control from the Insert panel and set a couple of properties on it by hand. Nothing is deleted that doesn't already need deleting (nothing does, as it turns out), and nothing needs renaming except the two brand-new controls in Part 2.

You already have ctnLinkModal_2 back (the version from before tonight's edits). That means the two real bugs from earlier — the display name field getting silently overwritten, and stale text showing up after Cancel — are still present in what you have right now. Part 1 below fixes those two things. Do Part 1 tonight regardless of anything else. Part 2 (visual restyle to match the rest of the screen) is lower priority — nice to have before the demo, not required for it to work correctly.

Part 1 — Fix the two real bugs ~10 minutes, zero risk

Every change below is a property edit on a control that already exists. No inserting, no deleting, no renaming.

1a. Display name field gets silently overwritten

What's wrong: the "Link Display Name" box falls back to a filename auto-guessed from the URL field whenever a context variable called DisplayNamePrefill changes — which happens live as you type the URL, even after you've already typed your own name in first.

Where to clickWhat to do
Open ViewItem → open any item → click Add Link (or Edit on an existing link) to open the modal → in the Tree view, expand into ctnLinkModal_2 → cardLinkModal_2 → frmLink_2 → LinkText_DataCard2_2 and click DataCardValue28_2 (the actual text input, not the data card or its label). In the Properties dropdown at the top of the formula bar, select Default. Select all in the formula bar (Ctrl+A), delete, and type exactly: 
Parent.Default

1b. URL field OnChange (the thing that was pushing the bad value in) — remove it

The Default fix above stops DataCardValue28_2 from listening to DisplayNamePrefill, but the URL field is still setting that variable on every keystroke for no reason now. Clean it out so nothing's left half-wired.

Where to clickWhat to do
Still inside the modal's Tree: frmLink_2 → LinkUrl_DataCard2_2, click DataCardValue29_2 (the URL text input). In the Properties dropdown, select OnChange. Select all in the formula bar (Ctrl+A) and press Delete, leaving it completely blank. Click away from the control to commit the blank formula.

1c. Stale text left over after Cancel, then Add

What's wrong: when you open the modal fresh (via Edit or Add), Power Apps doesn't automatically clear the text boxes just because the underlying record changed — you have to explicitly tell the form to refresh with ResetForm(frmLink_2). Only the Cancel button does this today. The three places that open the modal don't, so whatever was last typed/shown just sits there.

Where to clickWhat to do
On the links list, click the Edit icon (pencil) on any existing link row to select EditLink_2 in the tree. Select OnSelect in the Properties dropdown. Find the line Set(varShowLinkModal, true) at the end and add a new line directly above it reading: 
ResetForm(frmLink_2);
So the formula ends with: 
Set(varSelectedLink, LookUp(PolicyLinks, ID = ThisItem.ID));
ResetForm(frmLink_2);
Set(varShowLinkModal, true)
Select btnNewLink_1 (the "Add Document Link" pill button near the link list). Same fix: in OnSelect, find Set(varSelectedLink, Defaults(PolicyLinks)); followed by Set(varShowLinkModal, true), and insert the ResetForm line between them: 
Set(varSelectedLink, Defaults(PolicyLinks));
ResetForm(frmLink_2);
Set(varShowLinkModal, true)
Select btnAddLinkEmpty1 (the "Add Document Link" button shown when the link list is empty). This one has two spots with the same Set(varSelectedLink, Defaults(PolicyLinks)); Set(varShowLinkModal, true) pair inside its OnSelect — one near the top of the formula, one further down inside the nested save-then-open-modal branch. Add the same ResetForm(frmLink_2); line between the two Set(...) calls in both spots.
Test all three: (1) type a display name, then paste a SharePoint URL — the name you typed should stay put. (2) Edit a link, note its text, hit Cancel, then click Add — the Add form should be completely blank, not showing the link you just cancelled out of.

Part 3 — Link icons and "Open In App" don't recognise every Office file variant Real bug

What's wrong: the file-type detection only ever recognised doc, docx, xls, xlsx, ppt, pptx, pdf. Anything else -- template files (.potx, .dotx, .xltx), macro-enabled files (.docm, .xlsm, .pptm), or the older 97-2003 template/show variants (.dot, .xlt, .pot, .pps, .ppsx, .ppsm) -- falls through to the generic link icon and the "Open In App" button doesn't show at all for those, even though the file genuinely is a Word/Excel/PowerPoint document. This is all three fixed in the same three places, plus the colour bonus: the "Open In App" button now switches between Word blue, Excel green, and PowerPoint red/orange based on the file type, instead of always being the same blue.

All three changes below are property edits on controls that already exist -- nothing new to insert, nothing to rename, no paste-a-whole-control risk like Part 2.

3a. Named formula GetFileExt — recognise every variant

Where to click: in Studio's top menu, go to App (or the </> Formulas icon depending on your Studio layout) to open App formulas. Find the GetFileExt named formula. Select all its text and replace with the block below.
GetFileExt(LinkUrl: Text): Text = With(
    {FileParamStart: Find("file=", LinkUrl, 1)},
    With(
        {
            FileNamePart: If(
                FileParamStart > 0,
                With(
                    {AmpPos: Find("&", LinkUrl, FileParamStart + 5)},
                    Mid(LinkUrl, FileParamStart + 5, If(AmpPos > 0, AmpPos - (FileParamStart + 5), Len(LinkUrl) - (FileParamStart + 5) + 1))
                ),
                If(Find("?", LinkUrl, 1) > 0, Left(LinkUrl, Find("?", LinkUrl, 1) - 1), LinkUrl)
            )
        },
        Switch(
            true,
            Right(FileNamePart, 5) = ".docx", "docx",
            Right(FileNamePart, 5) = ".docm", "docm",
            Right(FileNamePart, 5) = ".dotx", "dotx",
            Right(FileNamePart, 5) = ".dotm", "dotm",
            Right(FileNamePart, 4) = ".doc", "doc",
            Right(FileNamePart, 4) = ".dot", "dot",
            Right(FileNamePart, 5) = ".xlsx", "xlsx",
            Right(FileNamePart, 5) = ".xlsm", "xlsm",
            Right(FileNamePart, 5) = ".xltx", "xltx",
            Right(FileNamePart, 5) = ".xltm", "xltm",
            Right(FileNamePart, 4) = ".xls", "xls",
            Right(FileNamePart, 4) = ".xlt", "xlt",
            Right(FileNamePart, 5) = ".pptx", "pptx",
            Right(FileNamePart, 5) = ".pptm", "pptm",
            Right(FileNamePart, 5) = ".potx", "potx",
            Right(FileNamePart, 5) = ".potm", "potm",
            Right(FileNamePart, 5) = ".ppsx", "ppsx",
            Right(FileNamePart, 5) = ".ppsm", "ppsm",
            Right(FileNamePart, 4) = ".ppt", "ppt",
            Right(FileNamePart, 4) = ".pot", "pot",
            Right(FileNamePart, 4) = ".pps", "pps",
            Right(FileNamePart, 4) = ".pdf", "pdf",
            ""
        )
    )
)
This is the same named formula used by BuildLinksForSource and RefreshActiveCache to populate FileExt on every link row -- fixing it here fixes icons everywhere those two are used, you don't need to touch them separately.

Part 4 — Icons reset to the generic globe after Save Real bug, do this one

What's wrong: this is the same root cause as Part 3, just a different set of formulas. colLinks (the collection every link row and its icon reads from) gets its file-type column (FileExt) computed correctly when you first open the item -- but when you hit Save or Save & Close on the item itself (not the link modal), the success handler repopulates colLinks from scratch without recomputing that column. Every icon's logic falls back to the generic globe the instant that happens, because there's no FileExt value to switch on any more. Closing and reopening the item "fixes" it only because that re-runs the correct population logic from scratch.

Three places do this -- the item's Save button, Save & Close button, and the "Add Document Link" button's save-then-open-modal branch (for a brand new item that hasn't been saved yet). All three need the exact same one-line fix.

Where to click: open ViewItem, open any item. You'll edit OnSelect on three different buttons in turn: btnSaveClose1, btnSave1, and btnAddLinkEmpty1. In each one's formula, find this exact line:

ClearCollect(colLinks, Filter(PolicyLinks, PolicyTrackerSource.Id = varSavedRecord.ID));

...and replace it with this line (adds the same FileExt column that the "open item" path already computes correctly):

ClearCollect(colLinks, AddColumns(Filter(PolicyLinks, PolicyTrackerSource.Id = varSavedRecord.ID), FileExt, GetFileExt(LinkUrl)));

That's the entire fix, done identically three times. Nothing else in either button's formula changes.

Test: open an item that already has a mix of Word/Excel/PowerPoint/PDF links, hit Save (not Save & Close) -- the icons should stay exactly as they were, not flash to the generic globe. Then try Save & Close, reopen, confirm still correct. Then try it on the "Add Document Link" first-save path on a brand new unsaved item.

On the icon load delay

You also mentioned the icons take a couple of seconds to actually render even when everything else on the row (text, buttons) appears instantly. I can't test this live, so this is my best honest read rather than a confirmed fix: word_icon/excel_icon/ppt_icon/pdf_icon aren't formulas, they're uploaded media assets in the app, and Power Apps fetches media assets over the network the first time each one is actually referenced in a session/screen load -- that's normal Player/Studio behaviour, not something introduced by any formula change here, and not something a Power Fx edit can speed up. The only lever I'm aware of (unverified, since I can't test it) is the file size of the actual icon images themselves -- if they're large PNGs, re-exporting them as small, compressed PNGs or SVGs in whatever tool you made them in might help. I'd treat this as a "live with it" cosmetic quirk rather than something worth spending more time on before tomorrow.
