18. Search-as-you-type and every badge/chip count re-query SharePoint directly PERFORMANCE

Screens: Main (search overlay, 4 dashboard badges, ~10 type-filter chip counts), Historic (search box, footer count).

These all run Filter('Policy Proof Tracker', ...) directly against the live SharePoint list, several using Lower(...)/in which SharePoint can't delegate — meaning past the delegation warning threshold (typically 500–2000 rows depending on your list settings), results and counts will silently under-count instead of erroring, and search re-hits the live source on every keystroke rather than filtering something already in memory.

Recommended approach (not a quick copy-paste — this is a real refactor, described here so it's not forgotten, worth doing as its own session rather than squeezed in tonight): build one ClearCollect of the currently-relevant rows once per real state change (tab switch, filter change), and have search/badges/chip-counts all read from that local collection via CountRows/Filter instead of re-querying 'Policy Proof Tracker' each time.
