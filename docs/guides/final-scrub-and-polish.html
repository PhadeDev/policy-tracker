Tier 3 — Naming drift

4. Duplicate-entry modal variable name drift: varShowDuplicateModal vs varShowDuplicateModal1 DRIFT

Screens: App.OnStart initializes Set(varShowDuplicateModal, false) (no suffix). ViewItem's actual duplicate-confirmation modal (DuplicateModal1.Visible, plus the Set() calls that open/close it) all read and write varShowDuplicateModal1 — with a "1" suffix, matching the naming convention nearly every other control on this screen uses (lblOwnerLabel1, conRowDateFlag1, etc.).

The two names never intersect: varShowDuplicateModal is set once in OnStart and never read again anywhere in the app; varShowDuplicateModal1 is read/written throughout ViewItem but never explicitly initialized. This mostly self-heals today — an unset variable evaluates to Blank(), which is falsy for Visible, so the modal still starts closed — but it's a real naming bug, not intentional design, and worth fixing before it causes a genuinely confusing symptom (e.g. if Visible's logic ever gets more complex than a bare read).

Fix — App → OnStart, one-line change:

ControlPropertyOldNew
App.OnStartSet(...) callSet(varShowDuplicateModal, false)Set(varShowDuplicateModal1, false)

Tier 4 — Performance, not urgent

All three are real, but none are wrong today — they're about maintenance risk and behaviour as the list grows. Safe to schedule as a fast-follow.

5. The file-extension/link-loading formula is copy-pasted six times across two screens MAINTENANCE

Screens: Main.yaml (release-row click, search-result-row click) and ViewItem.yaml (both branches of OnVisible, frmLink_2.OnSuccess, the delete-link confirm action) all contain the identical ~35-line block: ClearCollect(colLinks, AddColumns(Filter(PolicyLinks, PolicyTrackerSource.Id = ...), FileExt, With({FileParamStart: Find("file=", LinkUrl, 1)}, With({FileNamePart: ...}, Switch(true, Right(FileNamePart,5)=".docx","docx", ...))))).

Why it matters: any bug fix or new file-type support (e.g. adding .pdf variants, handling a URL shape that doesn't match) has to be applied in all six places by hand, or the six copies silently drift apart. Not wrong today — every copy currently matches — but it's the same shape of risk that caused the NewsletterPack sort-order duplication this pass fixed.

Recommended approach (a real refactor, not a copy-paste fix — worth its own session): if a Power Fx named formula is available in your Studio version, define this once at the app level and call it from all six sites; otherwise, keep it as a documented "these six must match" note until named formulas are available.

6. Main's type-filter chip bar re-scans the whole table per chip, on every render PERFORMANCE

Screen: Main — ChipLabel.Text, inside the ~10-row type-filter chip gallery. Each chip independently runs CountRows(Filter('Policy Proof Tracker', <4-branch state condition> && ownerFilter && 'Document Type'.Value = ThisItem.ChoiceValue)) — a fresh full-table filter per chip, up to 10 times, recomputed on every re-render (e.g. every time varSelectedState, varFilterOwner, or varTypeFilter changes).

Fix direction: compute per-type counts once (e.g. GroupBy/AddColumns into a collection keyed by document type, built at the same point varSelectedState changes) and have ChipLabel.Text just read the precomputed count instead of re-filtering.

7. Duplicate-entry Thursday picker re-filters 52 times instead of grouping once PERFORMANCE

Screen: ViewItem — btnDuplicate1.OnSelect. ForAll(Sequence(52), If(..., CountRows(Filter(colUpcomingEntries, Year(...)=Year(thurDate) && Month(...)=Month(thurDate) && Day(...)=Day(thurDate))))) runs a fresh Filter+CountRows over colUpcomingEntries once per iteration, 52 times, to build the list of upcoming Thursdays and their item counts.

Fix direction: pre-group colUpcomingEntries by date once (e.g. GroupBy) before the ForAll(Sequence(52), ...), then look counts up from the grouped collection instead of re-filtering the whole collection 52 times. Only fires when a user opens the duplicate flow, so not urgent, but it's 52x more work than needed every time it does.
