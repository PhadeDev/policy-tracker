Centralize Document Type Colours + Sort Order

You spotted this while editing Overview: the same document-type colour Switch is hand-copied across the app, so adding a document type (or recolouring one) means finding and editing every copy by hand. This consolidates every one of them down to a single definition in App.OnStart, referenced everywhere else via LookUp.

What this fixes, concretely: found 6 fully-duplicated colour/sort Switch blocks across the app (not counting the one canonical copy in OnStart itself) — 5 on a single control in Overview, 1 in Main, 2 in NewsletterPack (a duplicate-of-a-duplicate, since that screen already has the same formula copied twice for its own reasons), plus Historic's badge which uses a completely different, more limited colour mechanism and needs replacing outright rather than just pointed at the shared table. After this guide, changing a document type's colour or adding a new one is a one-place edit in OnStart.
Jump to: OnStart — the one source of truth (do this first) Overview — 5 duplicated switches on one control Main — sort-order switch NewsletterPack — sort-order switch, both copies Historic — replace the badge with a proper pill

OnStart — the one source of truth

1. Extend colChoiceColors with a faded variant and a sort rank

Right now each row only carries ChoiceValue and ColorCode. Adding two more fields to the same row means every consumer — solid colour, faded/hover colour, and sort order — all come from this one place. The faded values are the exact same RGB at 10% alpha, matching the fade you already hand-built in Overview's hover state (not ColorFade(), which shifts the colour rather than just its opacity — alpha-only keeps the faded version visually tied to its solid counterpart). Sort order matches the ranking already used identically in Main and NewsletterPack today (and documented in this guide's own index page).

App → OnStart, find ClearCollect(colChoiceColors, ...) and replace the whole thing with:

ClearCollect(
    colChoiceColors,
    ForAll(
        Choices('Policy Proof Tracker'.'Document Type'),
        {
            ChoiceValue: Value,
            ColorCode: Switch(
                Value,
                "📄 MoD Policy", RGBA(28, 32, 48, 1),
                "📗 Manual", RGBA(0, 130, 60, 1),
                "📄 CFI", RGBA(0, 100, 200, 1),
                "📄 CFSO", RGBA(130, 0, 180, 1),
                "📕 SOI", RGBA(190, 30, 30, 1),
                "📖 Regulation", RGBA(0, 150, 150, 1),
                "📄 CBN", RGBA(210, 100, 0, 1),
                "📄 ACSO", RGBA(0, 60, 130, 1),
                "📑 Internal Document", RGBA(90, 90, 110, 1),
                "📜 Newsletter Item", RGBA(160, 120, 0, 1),
                RGBA(150, 150, 150, 1)
            ),
            ColorCodeFaded: Switch(
                Value,
                "📄 MoD Policy", RGBA(28, 32, 48, 0.1),
                "📗 Manual", RGBA(0, 130, 60, 0.1),
                "📄 CFI", RGBA(0, 100, 200, 0.1),
                "📄 CFSO", RGBA(130, 0, 180, 0.1),
                "📕 SOI", RGBA(190, 30, 30, 0.1),
                "📖 Regulation", RGBA(0, 150, 150, 0.1),
                "📄 CBN", RGBA(210, 100, 0, 0.1),
                "📄 ACSO", RGBA(0, 60, 130, 0.1),
                "📑 Internal Document", RGBA(90, 90, 110, 0.1),
                "📜 Newsletter Item", RGBA(160, 120, 0, 0.1),
                RGBA(150, 150, 150, 0.1)
            ),
            SortOrder: Switch(
                Value,
                "📄 MoD Policy", 1,
                "📄 ACSO", 2,
                "📖 Regulation", 3,
                "📄 CFSO", 4,
                "📄 CFI", 5,
                "📄 CBN", 6,
                "📗 Manual", 7,
                "📜 Newsletter Item", 8,
                "📕 SOI", 9,
                "📑 Internal Document", 10,
                99
            )
        }
    )
);
Next time you add a document type in SharePoint: add one branch to each of these three Switches (colour, faded colour, sort rank) here in OnStart, nowhere else. Every screen below reads live from this collection.

Overview — 5 duplicated switches, one control

2. Button3 (the "Document Type" pill in each Thursday's inner list)

This one control currently has the full 10-branch colour switch copy-pasted five times — BorderColor, HoverBorderColor, HoverFill (faded), PressedBorderColor, PressedFill. All five collapse to one-line lookups.

Visual side-effect, not just a refactor: Overview currently has its own hand-typed palette that doesn't match OnStart's colChoiceColors (e.g. Regulation is a bright blue RGBA(62,142,222,1) here vs. teal RGBA(0,150,150,1) in the shared table). After this change, Overview's colours will shift to match Main/ViewItem exactly — which is presumably the point of centralizing, but it is a visible colour change on this screen, not a no-op.
PropertyNew value (full replacement)
BorderColor=Coalesce(LookUp(colChoiceColors, ChoiceValue = ThisItem.'Document Type'.Value, ColorCode), RGBA(102,102,102,1))
HoverBorderColor=Coalesce(LookUp(colChoiceColors, ChoiceValue = ThisItem.'Document Type'.Value, ColorCode), RGBA(102,102,102,1))
HoverFill=Coalesce(LookUp(colChoiceColors, ChoiceValue = ThisItem.'Document Type'.Value, ColorCodeFaded), RGBA(102,102,102,0.1))
PressedBorderColor=Coalesce(LookUp(colChoiceColors, ChoiceValue = ThisItem.'Document Type'.Value, ColorCode), RGBA(102,102,102,1))
PressedFill=Coalesce(LookUp(colChoiceColors, ChoiceValue = ThisItem.'Document Type'.Value, ColorCode), RGBA(102,102,102,1))

Each of those was previously ~11 lines (a full Switch) — now one line each, 5 lines total instead of ~55.

Button3.Color (the black/white text-contrast switch for readability on light vs dark fills) is left alone — it only exists in this one place, isn't duplicated anywhere else, so it isn't part of the maintenance problem this guide is solving. No need to touch it.

Main — sort-order switch

3. ReleaseGallery.Items — the per-item sort within each Thursday group

Full replacement:

=Sort(
    ThisItem.GroupedItems,
    LookUp(colChoiceColors, ChoiceValue = 'Document Type'.Value, SortOrder),
    SortOrder.Ascending
)

Was a 27-line, 20-branch Switch (10 types, each listed twice for emoji + plain-text fallback) — now 4 lines. colChoiceColors only carries the emoji-prefixed keys, matching what 'Document Type'.Value actually returns from SharePoint (the plain-text branches in the old switch were dead defensive code that never matched anything real).

NewsletterPack — sort-order switch, both copies

Do this after the Consultation Only guide if you're doing both — the formulas below already include that guide's exclusion line ('External consultation required' = false). 4. Screen OnVisible, full replacement

=Set(varNewsletterTargetDate, DateAdd(Today(), Mod(5 - Weekday(Today()) + 7, 7), "Days"));
Refresh('Policy Proof Tracker');
ClearCollect(
    colNewsletterPack,
    Sort(
        AddColumns(
            Filter(
                'Policy Proof Tracker',
                Year('Planned Publish Date') = Year(varNewsletterTargetDate) &&
                Month('Planned Publish Date') = Month(varNewsletterTargetDate) &&
                Day('Planned Publish Date') = Day(varNewsletterTargetDate) &&
                (
                    ReadyForRelease = true ||
                    'Approved For Release By'.Value = "XO" ||
                    'Approved For Release By'.Value = "ACOS" ||
                    'Approved For Release By'.Value = "DCOMD" ||
                    'Approved For Release By'.Value = "D Commander"
                ) &&
                Published = false &&
                'External consultation required' = false
            ),
            SortKey,
            LookUp(colChoiceColors, ChoiceValue = 'Document Type'.Value, SortOrder),
            IsFinalApproved,
            Or(
                'Approved For Release By'.Value = "XO",
                'Approved For Release By'.Value = "ACOS",
                'Approved For Release By'.Value = "DCOMD",
                'Approved For Release By'.Value = "D Commander"
            ),
            PackTitle,
            Coalesce('Title for Newsletter', Title),
            PackAudience,
            Coalesce(Audience, ""),
            PackSummary,
            Coalesce('Item summary', "")
        ),
        SortKey,
        SortOrder.Ascending
    )
)

Historic — replace the badge with a proper pill

6. BadgeCanvas1_1 uses the Badge control's built-in theme enum — only 7 colours exist, already all claimed, can't be extended

This is why it couldn't just get a LookUp pointed at it like the others: Badge's ThemeColor only accepts a fixed set of named values (Success/Informative/Important/Danger/Brand/Severe/Warning), all 7 already assigned across the 10 real document types (with some types doubled up on the same colour, e.g. CBN and Internal Document both get Severe). Replacing the control type entirely with the same "solid border/text, faded fill" pill already used for LinkCount on Main fixes both problems at once — unlimited colours, and it's driven by the same shared table as everywhere else.

Delete BadgeCanvas1_1, add a new ModernText@1.0.0 in its place named PillDocType1:

PropertyValue
Align=Align.Center
BorderColor=Coalesce(LookUp(colChoiceColors, ChoiceValue = ThisItem.'Document Type'.Value, ColorCode), RGBA(150, 150, 150, 1))
BorderStyle=BorderStyle.Solid
BorderThickness=1
Color=Coalesce(LookUp(colChoiceColors, ChoiceValue = ThisItem.'Document Type'.Value, ColorCode), RGBA(150, 150, 150, 1))
Fill=Coalesce(LookUp(colChoiceColors, ChoiceValue = ThisItem.'Document Type'.Value, ColorCodeFaded), RGBA(150, 150, 150, 0.1))
Font=Font.Lato
FontWeight=FontWeight.Bold
Height=26
RadiusTopLeft / RadiusTopRight / RadiusBottomLeft / RadiusBottomRight=13 (half the height — full pill/capsule shape)
Size=11
Text=ThisItem.'Document Type'.Value
Visible=!IsBlank(ThisItem.'Document Type'.Value)
Width=210
X=txtTaskDateHeader_3.X
Y=17

ModernText@1.0.0 is the same control type LinkCount on Main already uses for an identical solid-border/faded-fill/coloured-text pill — confirmed-working combination in this app, not a new pattern.
