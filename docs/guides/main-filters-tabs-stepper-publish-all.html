Fix 1 — the growing gap under each date group

Screen: Main. This is a real formula bug, not a setting you're missing. Container1 (the per-date-group container, direct child of HeaderGallery) has:

Height: CountRows(ThisItem.GroupedItems)*130

and its child ReleaseGallery has:

Height: CountRows(ThisItem.GroupedItems)*110
Y: 40

Do the maths on those two together and the leftover empty space at the bottom of Container1 works out to (N × 20) - 40, where N is how many items are in that date group. For a 2-item group that's exactly 0 (flush, looks fine). For a 5-item group — like your "Thursday, 25th June" one — that's 60px of dead space before the next group starts. The gap gets worse the more items are in a group, which is exactly the symptom you're seeing. The *130 was a rough per-card estimate that never accounted for the fixed Y=40 offset being a one-time cost, not a per-card one.

The fix

Replace Container1.Height with a formula that has zero leftover for any group size:

40 + CountRows(ThisItem.GroupedItems)*110

The 20px breathing room between separate date groups is already handled by HeaderGallery.TemplatePadding=20 — you don't need to add anything for that here, this fix is purely about not over-allocating space inside a single group.

Formula audit — verifying "12" and "12" aren't the same formula

Fair thing to be suspicious of. Checked both directly against your export — they're genuinely different conditions that happen to produce the same number right now, not a copy-paste:

// Future Releases tab count (Count_2)
CountRows(
    Filter(
        'Policy Proof Tracker',
        'Planned Publish Date' > DateAdd(Today(), 28, "Days")
    )
)

// Previously Published tab count (Count_1)
CountRows(
    Filter(
        'Policy Proof Tracker',
        'Planned Publish Date' >= DateAdd(Today(), -28, "Days") &&
        'Planned Publish Date' < Today() &&
        Published = true
    )
)

One counts anything more than 4 weeks in the future with no Published condition at all; the other counts things published in the last 4 weeks specifically. Both landing on 12 today is a coincidence of your current data, not a bug. Also checked Due Next 4 Weeks (already fixed to the correct >=Today() && <=+28 days logic, showing 5 correctly) and Issues (<Today() && Published=false, showing 0 correctly) — all four tabs are using four genuinely distinct, correct conditions.
