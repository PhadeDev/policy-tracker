1. Saving is blocked on existing items due today or soon BLOCKING

Do this one first, it's the only thing on this page that stops real, everyday work. The 10-day warning banner correctly only shows for brand new entries, but the Save button runs its own separate copy of the same date check, and that copy was never told to skip existing items. Right now, opening any existing entry whose Planned Publish Date happens to be within 10 days of today and clicking Save just refuses, no way around it except ticking High Priority.

Three buttons on ViewItem have their own copy of this check: btnSave1, btnSaveClose1, and btnAddLinkEmpty1 (the auto-save that runs the first time you add a link to an item). Same fix in all three.

Find this exact condition inside each button's OnSelect:

!IsBlank(dpkPublishDate1.SelectedDate) &&
DateDiff(Today(), dpkPublishDate1.SelectedDate, "Days") >= 0 &&
DateDiff(Today(), dpkPublishDate1.SelectedDate, "Days") < 10 &&
!ckbHighPriority1.Value,

Replace with (one new line added, everything else identical):

varSelectPolicy <> "Filled" &&
!IsBlank(dpkPublishDate1.SelectedDate) &&
DateDiff(Today(), dpkPublishDate1.SelectedDate, "Days") >= 0 &&
DateDiff(Today(), dpkPublishDate1.SelectedDate, "Days") < 10 &&
!ckbHighPriority1.Value,

Same guard the warning banner already correctly uses. Existing items can now always be saved no matter how close the date is, the 10-day rule only blocks a genuinely new entry, or a release date being changed on something that was never saved yet.

Test: open an item that already exists (not one you just created), due today or within the next 10 days, change something small, click Save & Close, confirm it saves instead of showing the warning.

2. Duplicate record race on Add Document Link DATA LOSS RISK

btnSave1 and btnSaveClose1 both disable themselves the instant a save starts, so clicking either one twice in a row is safe. btnAddLinkEmpty1 (the "Add Document Link" button shown before any links exist) runs the exact same create-or-update logic but was never given the same disable guard.

Failure scenario: on a brand new unsaved entry, click Save & Close, then before that finishes click Add Document Link. The first save hasn't updated the record's ID yet, so the second click also creates a brand new record, same values, two near-identical entries.

btnAddLinkEmpty1, add this one property, everything else on it stays as is:

DisplayMode: =If(varSaving, DisplayMode.Disabled, DisplayMode.Edit)

Same property, same variable, same value btnSave1/btnSaveClose1 already use, so all three now go disabled together the instant any one of them starts a save.

Test: on a brand new unsaved entry, click Save & Close, then immediately click Add Document Link before the notification appears. It should now be disabled the whole time the save is running.

3. Publish All patches the wrong record WRONG DATA, STAFF-FACING

Main.yaml, control PublishAllBtn. Its LookUp('Policy Proof Tracker', ID = ID) is ambiguous, both sides of that comparison resolve to the same innermost scope, so it always matches "itself" and returns the same one record every time through the loop, regardless of which item is actually meant to be published.

Failure scenario: click "Mark All as Published" on a Thursday with 3 items, the same one record gets patched 3 times, the other two are left unpublished, while the notification claims all 3 succeeded.

PublishAllBtn.OnSelect, full replacement (only the ForAll/LookUp lines change):

=// PublishAllBtn.OnSelect
Set(varPublishGroupDate, ThisItem.PlannedDate);
Set(varPublishGroupCount, CountRows(Filter(ThisItem.GroupedItems, 'External consultation required' = false)));
ForAll(
    Filter(ThisItem.GroupedItems, 'External consultation required' = false) As PubItem,
    Patch(
        'Policy Proof Tracker',
        LookUp('Policy Proof Tracker', ID = PubItem.ID),
        {
            Published: true,
            'Published Date': varPublishGroupDate
        }
    )
);
Refresh('Policy Proof Tracker');
Notify(
    "Published all " & varPublishGroupCount & " items for " & Text(varPublishGroupDate, "d mmm yyyy"),
    NotificationType.Success
)

Naming the outer loop item As PubItem means the bare ID inside LookUp unambiguously refers to the LookUp's own scope while PubItem.ID refers to the outer loop item, no shadowing, same fix shape the correctly-written single-item BossPublishBtn already uses. (Note: an earlier version of this guide used 'Policy Proof Tracker'.ID = ThisRecord.ID, which is invalid, that syntax evaluates to the whole ID column as a Table rather than disambiguating a scope, causing a "Table, Number" type error — As PubItem is the correct fix.)

Test: pick a Thursday group with at least 2 unpublished items, click Mark All as Published, then reopen each item and confirm every one shows Published, not just one of them.

4. Closing without saving silently discards edits DATA LOSS

Every editable field on ViewItem sets varUnsavedChanges to true on change, over 20 controls do this. That variable is never actually read anywhere. btnClose1 navigates away unconditionally, so a user who edits something and clicks Close instead of Save loses it with zero warning.

Fastest, lowest risk fix reuses the confirm modal you already have (DeleteItemModal1) rather than building a new one control by control.

Step by step:

  1. In the Tree view, right-click DeleteItemModal1, choose Duplicate. Studio auto-renames the copy and everything nested inside it, note what it actually named things, the rest of these steps refer to "the duplicated <control>" for that reason.
  2. Duplicated top-level container (was DeleteItemModal1): Visible to =locDisplayDiscardPopUp.
  3. Duplicated close icon button (was btnCloseModal1): OnSelect to =UpdateContext({locDisplayDiscardPopUp: false}).
  4. Duplicated Cancel button (was btnCancelModal1): same as above, OnSelect to =UpdateContext({locDisplayDiscardPopUp: false}).
  5. Duplicated title text (was txtModalTitle1, said "Delete"): Text to ="Discard changes?".
  6. Duplicated body text (was txtModalText1): Text to ="You have unsaved changes. Are you sure you want to close without saving?".
  7. Duplicated red confirm button (was btnModalAction1): replace OnSelect with the formula below, this is the only genuinely new logic, it's exactly what btnClose1.OnSelect already does today, just now behind a confirm step. Also change its Text to ="Discard".
=UpdateContext({locDisplayDiscardPopUp: false});
Set(varSelectPolicy, "Empty");
Set(varSelectedRelease, Blank());
Set(varUnsavedChanges, false);
Set(varShowLinkModal, false);
If(varSourceScreen = "Historic", Navigate(Historic, ScreenTransition.Fade), Navigate(Main, ScreenTransition.Fade))

btnClose1.OnSelect, full replacement:

=If(
    varUnsavedChanges,
    UpdateContext({locDisplayDiscardPopUp: true}),
    Set(varSelectPolicy, "Empty");
    Set(varSelectedRelease, Blank());
    Set(varUnsavedChanges, false);
    Set(varShowLinkModal, false);
    If(varSourceScreen = "Historic", Navigate(Historic, ScreenTransition.Fade), Navigate(Main, ScreenTransition.Fade))

No unsaved changes, Close behaves exactly as today, instant, no popup. Unsaved changes present, it stops and asks first.

5. Deep link shows a blank form if the item isn't found CAN'T TEST WITHOUT PUBLISHING

ViewItem.OnVisible's deep link branch has no else case. If the linked item ID doesn't resolve to a real record (deleted, bad link, stale bookmark), the screen silently falls through to what looks exactly like a blank new-entry form, no message, real risk someone starts filling it in as new without realising the link failed.

Find this exact block in ViewItem.OnVisible:

If(
    !IsBlank(varDeepLinkItem),
    Set(varSelectedRelease, varDeepLinkItem);
    Set(varSelectPolicy, "Filled");
    Set(varParentSaved, true);
    Set(varSelectedItem, varDeepLinkItem);
    Set(varSourceScreen, "Main");
    Set(varSelectedPerson, varDeepLinkItem.Owner);
    Refresh(PolicyLinks);
    BuildLinksForSource(varDeepLinkItem.ID);
    Set(varOwnerDefault, Office365Users.SearchUserV2({searchTerm: varDeepLinkItem.Owner.Email, isSearchTermRequired: true, top: 1}).value)
);

Replace with (only the last line changes, from a closing bracket to a comma plus a new else branch):

If(
    !IsBlank(varDeepLinkItem),
    Set(varSelectedRelease, varDeepLinkItem);
    Set(varSelectPolicy, "Filled");
    Set(varParentSaved, true);
    Set(varSelectedItem, varDeepLinkItem);
    Set(varSourceScreen, "Main");
    Set(varSelectedPerson, varDeepLinkItem.Owner);
    Refresh(PolicyLinks);
    BuildLinksForSource(varDeepLinkItem.ID);
    Set(varOwnerDefault, Office365Users.SearchUserV2({searchTerm: varDeepLinkItem.Owner.Email, isSearchTermRequired: true, top: 1}).value),
    Notify("This item could not be found. It may have been deleted or the link may be incorrect.", NotificationType.Error);
    Set(varDeepLinkNotFound, true)
);

Important: Studio blocks Navigate inside a screen's own OnVisible outright, even nested inside an If that only reaches it conditionally — it can't statically prove the navigate won't fire unconditionally, so it refuses the whole property. The formula above only sets a flag, varDeepLinkNotFound; the actual navigate happens from a one-shot Timer instead, which is the standard workaround for this restriction.

Add a Timer control (Insert → search "Timer"), rename it tmrDeepLinkNotFound in Tree view, and set:

  • Visible: =false
  • AutoStart: =false
  • Repeat: =false
  • Duration: =100
  • Start: =varDeepLinkNotFound
  • OnTimerEnd: =Navigate(Main, ScreenTransition.Fade); Set(varDeepLinkNotFound, false)

Setting varDeepLinkNotFound to true starts the timer, which fires once ~100ms later and does the real navigate outside of OnVisible, then resets the flag so it's armed for next time.

Test without publishing: temporarily change Value(varItemID) to something you know doesn't exist (e.g. 999999), open ViewItem directly, confirm you get the "could not be found" notification and land back on Main a moment later instead of a blank form. Revert the temporary change once confirmed.

11. Owner shows current user instead of real owner CONFIRMED ROOT CAUSE

Found live: opening any item on ViewItem shows the current logged-in user as Owner regardless of which item was clicked, every other field on the record (Title, Document Type, links) displays correctly.

Real cause, confirmed by reading ViewItem.OnVisible directly: the tail end of that formula explicitly calls Reset() on sixteen input controls — txtTitle1, drpDocType1, dpkPublishDate1, ckbHighPriority1, and so on through every field on the form — specifically so each one re-reads its Default/DefaultSelectedItems against the newly-selected item every time the screen opens. cmb_Owner1, the Owner people-picker, is the one control missing from that list. In this control library, Default/DefaultSelectedItems only evaluates once, the moment the control is first created — it does not automatically re-read the underlying variable on every subsequent visit unless something forces it to with Reset(). Same root idea as the dpkPublishDate1 Reset(Self) incident already in the Bible, just the opposite failure mode: that one fired a Reset it shouldn't have, this one is missing a Reset it needs. Once cmb_Owner1 first rendered, it locked onto whatever it showed then and never updated again, regardless of which item you clicked into afterwards — matches "doesn't matter what I click, it's always me" exactly, not something timing-dependent.

Reset(txtReasonHighPri1);
Reset(ckbExtConsult1);
Reset(txtExtConsultDetails1);
Reset(ckbTopItem1);
Reset(txtNewsletterTitle1);

Reset(txtSummary1);
Reset(txtRemarks1);
